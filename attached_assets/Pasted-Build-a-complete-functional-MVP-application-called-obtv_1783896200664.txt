Build a complete, functional MVP application called obtv-ai, a fully local AI-powered media intelligence and semantic video search platform.

This is inspired by platforms that automatically understand video libraries, but it must be an original implementation. Do not copy proprietary branding, code, interface designs, or product language.

The finished application will run on an on-premises Ubuntu server with:

NVIDIA RTX PRO 6000 Blackwell
96 GB VRAM
NVIDIA CUDA-capable drivers
NVIDIA Container Toolkit
Docker Engine
Docker Compose
Local SMB, NFS, or directly mounted media storage
No cloud AI APIs
No media or metadata sent outside the local network

Replit will be used to develop the application and repository. The deployed application must run through Docker Compose on the local GPU server.

Primary product goal

A user should be able to place a video file into a watched media folder. The system must automatically:

Discover the file.
Verify that the file has finished copying.
Extract technical metadata with ffprobe.
Create a browser-compatible proxy.
Extract audio.
Transcribe speech with timestamps.
Perform speaker diarization.
Detect video scenes.
Extract representative scene frames.
Generate visual embeddings.
Generate short scene descriptions.
Detect and cluster faces.
Generate searchable embeddings.
Store all results locally.
Make the media searchable from a browser.
Let the user play search results at the matching timecode.
Let a local AI agent answer questions about indexed media.
Let the user generate and export clip lists.

The minimum successful workflow is:

Upload or discover video
→ process video
→ transcribe it
→ index transcript segments
→ search using natural language
→ display matching segments
→ play the video from the exact matching timecode

Non-negotiable requirements
Build real working code, not static mockups.
Do not use OpenAI, Anthropic, Google, Azure, AWS, or other cloud AI services.
All AI inference must run locally.
Do not require internet access after initial model downloads.
Use Docker Compose for deployment.
Use PostgreSQL for relational metadata.
Use Qdrant for vectors.
Use Redis for queues and worker coordination.
Use FastAPI for the backend.
Use React with TypeScript for the frontend.
Use FFmpeg and ffprobe for media processing.
Use NVIDIA GPU acceleration where appropriate.
Store original media outside the application containers.
Mount original media read-only by default.
Store proxies, thumbnails, extracted audio, and AI artifacts separately.
Never modify or delete source media.
Every AI result must resolve to an asset ID and start/end timecode.
AI answers about media must cite the source asset and timecode.
Long-running processing must occur in background workers.
All processing jobs must expose status, progress, logs, retries, and errors.
The app must recover cleanly after container or server restarts.
Do not store binary video data inside PostgreSQL or Qdrant.
